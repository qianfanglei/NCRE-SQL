# 第一章 数据库应用系统开发方法
>- 数据库系统（Database System，DBS）主要提供应用数据的组织、存储、维护、访问等数据管理功能。
>- 数据库应用系统（Database Application System）不仅为用户提供数据管理功能，还根据具体应用领域业务规则，通过数据库应用程序，实现了更为复杂的数据处理功能。
>- DBAS设计与开发是指根据具体应用领域数据管理和处理需求，设计应用数据在数据库中的组织和存储方式，即 **设计数据库模式或结构**，并根据应用领域业务规则设计数据库应用软件。
>- 本章重点围绕DBAS生命周期模型简要介绍 **DBAS设计、开发和运行维护各阶段的主要目标、工作内容、输出结果和采用的关键技术**。

## 1 数据库应用系统生命周期  

> 用现代工程的概念管理软件生产与开发全过程的典型方法有：  

> - 瀑布模型
> - 快速原型模型
> - 螺旋模型 **等**

- 软件工程与软件开发方法：
	- 瀑布模型  
	  　将软件开发的全过程定义为六个阶段：  
		- 项目规划  
		- 系统分析  
		- 总体设计  
		- 详细设计  
		- 编码调试与集成测试  
		- 运行维护阶段  
		
		　瀑布模型要求必须按照模型定义的顺序，由于各阶段活动行程一个线性顺序，便于大型复杂系统中多学科开发小组成员之间对于一些共性问题的协调，保证阶段工作的有效性、一致性和完备性。  
	  　然而这种文档驱动、强调阶段工作完备性的知道思想，在实际项目开发中往往难以完全保证。
	- 快速原型模型  
		　指导思想是不必把前七个阶段的活动做得尽善尽美后才启动下阶段的活动，先构建目标系统的一个可运行的早期版本，在用户的参与下经过不断的改进完善，最终得到满意的目标系统。分为以下几个阶段：
		- 快速分析
		- 设计构造原型
		- 运行原型
		- 评价原型
		- 改进原型，此步骤后重复步骤3-5

	- 螺旋模型  
		　这个模型将软件开发的全过程划分为四个阶段，也成为四个象限：
		- 项目规划
		- 风险评估
		- 工程实现
		- 用户评估  
		
		　开发过程每经过一个迭代周期，螺旋线就增加一圈，系统又生成一个新版本，迭代到用户对产品完全满意为止。

- DBAS生命周期模型
	- 项目规划
	- 需求分析
	- 系统设计
	- 实现与部属
	- 运行与维护

## 3 需求分析  

用户对DBAS在 **功能、性能、行为、设计约束** 等方面的期望和要求；  
需求分析过程由 **需求获取、需求分析、需求描述与规范说明、需求验证** 等步骤组成。

- 数据需求分析
	- 分析与辨识应用领域所管理的各类数据项和数据结构，形成数据字典的主要内容。
		- 数据项
		- 数据结构
		- 数据流
		- 数据存储
		- 处理过程
- 功能需求分析
	- 数据处理需求分析（事务规范）
		- 事务名称
		- 事务描述
		- 事务所访问的数据
		- 事务用户
	- 业务规划需求分析
- 性能需求分析  
	- **性能指标：**
		- 数据操作响应时间，或数据访问响应时间
		- 系统吞吐量
		- 允许并发访问的最大用户数
		- 每TPS代价值
	- **影响因素：**
		- 系统硬件资源
		- 网络通信设备性能
		- 操作系统环境
		- 数据库的逻辑设计和物理设计质量
		- DBMS的配置和性能
		- 数据库应用程序自身
- 其他需求分析
	- 存储需求分析
		- 初始数据库大小
		- 数据库增长速度
	- 安全性需求分析
		- DBAS系统应达到的安全控制级别
		- 各类用户的数据视图和视图访问权限
		- DBAS应有的口令保护机制或其他安全认证机制
	- 备份和恢复需求分析
		- DBAS运行过程中备份数据库的时间和备份周期
		- 所需备份的数据是全部数据库还是其中的一部分
		- 备份方式采用完全备份还是差异备份

## 4 系统设计

数据库应用系统设计包括 **概念设计、逻辑设计、物理设计** 三个步骤  

- 概念设计
	- 数据库概念模型设计
	- 系统总体设计
		- DBAS体系结构设计
		- DBAS系统硬件平台的选型和配置
		- 应用软件结构设计
		- 对需求分析阶段识别出的业务规则进行初步设计，细化业务规划流程，分析所处理的业务数据和处理方式，明确采用的关键技术和算法等
		- 对系统采用的关键技术进行方案选型和初步设计
- 逻辑设计
	- 数据库逻辑结构设计
	- 应用程序概要设计
	- 数据库事务概要设计
- 物理设计
	- 数据库物理结构设计
	- 数据库事务详细设计
	- 应用程序详细设计

# 第二章 需求分析

## 1 需求分析

软件产品的下列特性常常使得需求的获取困难重重：  

- 软件功能复杂
- 需求的可变性
- 软件产品的不可见性

需求分析的主要任务是 **分析清楚当前系统的业务流程**。

通常采用下列几种方法来获取需求：  

- 面谈
- 实地观察
- 问卷调查
- 查阅资料

需求分析过程：

- 标识问题
- 建立需求模型
- 描述需求
	- 需求描述是对待开发系统从宏观和整体上的一个完整描述。它精确地定义和说明了系统做什么以及交付的目标产品的约束条件，为软件生命周期中后续的活动提供了工作的依据和蓝图，是项目开发方和使用者或用户方的一个约定，也是项目后期审核和验收的依据。
	- 需求概述
	- 功能需求
	- 信息需求
	- 性能需求
	- 环境要求
	- 其他需求
- 确认需求（审核下列内容：）
	- 功能需求
	- 数据需求
	- 性能
	- 数据管理
	- 其他需求

## 2 需求分析方法

广为使用的结构化分析与功能建模方法主要有DFD、IDEF0等。

结构化分析方法的基本特征是 **抽象** 和 **分解**。

### DFD需求建模方法
也被称为过程建模和功能建模方法。核心是 **数据流**

DFD建模方法首先抽象出具体应用的主要业务流程，然后分析其输入。

其基本元素有四个：

- 数据流
- 处理
- 数据存储
- 外部项（数据源、数据重点）

DFD图采用自顶向下逐步细化的结构化分析方法表示目标系统。

### 其他需求建模方法
IDEF0方法简介：
>描述系统功能及相互关系，是一个有控制地逐步地展开细节，精确、准确、全面地描述系统。  
>元素只有两个：**箭头** 和 **活动**

UML用例模型简介：
>UML方法采用面向对象思想建模，其中的用例模型用于描述系统功能需求，用例图由系统、角色和用例三种模型元素及其之间的关系构成。

# 第三章 数据库结构设计


目标：  

1. 定义和描述应用领域涉及的数据范围
2. 获取应用领域或问题领域的信息模型
3. 描述清楚数据的属性特征
4. 描述清楚数据之间的关系

## 数据库概念设计

### 概念设计的依据及过程

**依据** 是 **需求分析阶段的文档** 包括：

- 需求说明书
- 功能模型
- 应用领域或问题域中的各类报表

通过这些文档的分析与理解，从中抽象出**数据的结构、语义、数据之间的相互联系及数据满足的约束条件**

**过程** 是：

- 明确建模目标
- 定义实体集（数据库概念设计阶段通常采用自底向上的方式）
- 定义联系（直接联系而非间接联系）
- 建立信息模型
- 确定实体集属性
- 对信息模型进行集成和优化

### 数据建模方法

- ER（实体联系）建模方法
- IDEF1X建模方法
